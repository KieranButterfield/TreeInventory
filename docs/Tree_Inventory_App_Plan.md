# Tree Inventory App — Project Plan & Agent Architecture

**Prepared for:** Kieran Butterfield
**Context:** LiDAR-based tree measurement app, using the Dec 2025 Newark Parks & Rec Significant Tree Inventory as a structural and methodology reference

---

## Executive Summary

This plan covers an iOS app that lets multiple surveyors, working at the same time on separate devices, measure tree DBH, height, and crown spread — DBH and crown spread via the LiDAR sensor in an iPhone 12 Pro+ / iPad Pro (2020+) camera, height via a digital tangent-angle tool, with manual override always available on any of the three. Work is organized into projects, each with its own list of sites, with a Projects screen as the app's home. Surveyors are assumed to have WiFi access in the field, so a shared Supabase backend is the live source of truth for who's measured what, letting surveyors see each other's progress in real time and avoid two people measuring the same tree. Records also export to CSV, matching the Newark Parks & Rec dataset's column layout (Section 1) for email/sharing. The app's screen structure is modeled directly on a reference app (TRAILVision), substituting trees for cameras and condition/DBH/height for battery/status metrics — including its Team tab, which becomes the multi-surveyor coordination view. The app itself should be built with Claude Code on a Mac with Xcode — not in this planning session — using the agent architecture in Section 7.

## 1. Reference Dataset: Newark Parks & Rec

This pilot uses the Dec 2025 Newark Parks & Rec inventory ([technology-service-center.github.io/Newark-Parks-and-Rec-Tree-Inventory](https://technology-service-center.github.io/Newark-Parks-and-Rec-Tree-Inventory)) as a template for data structure and field methodology, not as a set of trees to re-measure: 158 significant trees across 11 locations — 8 trail sections (JH-W, JH-P, JH-A, JH-L, JH-K, JH-R, JH-UD, JH-E) plus Phillips, Lewis, and Kells parks — recorded with the EpicCollect5 mobile app. The columns below are the **exact raw export headers** — the literal field names in the underlying data sheet, not the friendlier labels the website's dashboard displays for the same data:

| Field (raw export header) | Method |
|---|---|
| `ec5_uuid` | Auto-generated entry ID (UUID), e.g. `cde11e5b-4a35-489f-9873-c4f8a7886092` |
| `created_at` | Auto-generated capture timestamp |
| `uploaded_at` | Auto-generated sync/upload timestamp (separate from `created_at`) |
| `title` | Auto-generated, duplicates `ec5_uuid` — not a human-entered title |
| `1_Surveyor_Name` | Manual entry |
| `lat_2_Location` / `long_2_Location` | Phone GPS (decimal degrees) |
| `accuracy_2_Location` | Auto-generated GPS accuracy, in meters |
| `UTM_Northing_2_Location` / `UTM_Easting_2_Location` / `UTM_Zone_2_Location` | Auto-generated, converted from lat/long |
| `3_Park_Location` | Selected from the current project's site list |
| `4_Tree_ID` | Manual entry |
| `5_Photo` | iPhone or iPad camera, stored as a hosted URL |
| `6_Diameter_at_Breast` | Diameter tape at 4'4", recorded as circumference (inches) |
| `7_Tree_Height_ft` | Tangent height gauge + measuring wheel |
| `8_Tree_Spread_tip_to` / `9_Tree_Spread_tip_to` | Measuring wheel, two perpendicular readings (Sample 1 / Sample 2) |
| `10_Tree_Condition` | Visual rating: Good / Fair / Poor |
| `11_Notes_eg_mushroom` | Free text — also where multi-branch is flagged, rather than its own column |
| `Species` | Manual entry — new capture field, populating a column that's existed but been blank in every prior record |

Trees are classified **large mature** (DBH ≥ 20 in — 72.15% of the inventory), **young** (support removed), or **newly planted** (support still present). `ec5_uuid`, `created_at`, `uploaded_at`, and `title` are all auto-generated by EpicCollect5, not surveyor-entered, and `title` simply duplicates `ec5_uuid`. The website's dashboard list view shows a friendlier version of this same data (View, Title, Created At, Surveyor Name, Location, Park Location, Tree ID, Photo, and so on) — that's a display convenience, not the export format; `View` in particular is a dashboard-only link that opens a detail panel and isn't an export column at all. This plan matches the raw export headers directly, since that's the format the underlying data actually uses. The app computes `accuracy_2_Location` and the three `UTM_*` fields from phone GPS at capture time (Section 5.2), so those columns are populated rather than left blank. `Species` is new functionality: existing records leave it blank, but the app adds species as something surveyors capture going forward. Raw data lives in the [EpicCollect5 export Google Sheet](https://docs.google.com/spreadsheets/d/1CrADExohsLWq7vGZhaonv0nHoEUQfKkVnnjnETOM3Os/edit?gid=1277814094#gid=1277814094). EpicCollect5 itself isn't integrated with — it's reference only, since this app replaces it — so the app doesn't read from or write to it; it just matches the column layout closely enough that exported CSVs merge cleanly into the existing 158-tree dataset.

## 3. Existing Code: What It Already Does

The provided SwiftUI/ARKit/RealityKit sketch is a working start on the DBH tool specifically: `ARViewContainer` runs `ARWorldTrackingConfiguration` with `.meshWithClassification` scene reconstruction; a tap raycasts into the LiDAR mesh; nearby vertices within a 2 cm vertical slice and 40 cm radius of the tap are collected; a Kasa least-squares circle fit turns that slice into a radius → circumference; the result is shown as a status banner and as 3D text anchored in space. This becomes the AR-Capture module in Section 7.

## 5. App Design

### 5.1 Measurement tools

**Capture sequence.** The four steps run in a fixed order in the field, matching how a surveyor already moves around a tree: stand at sighting distance and capture height (sighting + photo), walk to the trunk for DBH, walk out to the canopy for crown spread, then enter species. The Capture screen (Section 5.3) guides surveyors through these four steps in order, each one confirmable or editable before moving to the next.

**1. Height — tangent-angle tool, with manual override.** This is the digital equivalent of the tangent height gauge surveyors already use: stand a known horizontal distance from the tree, sight the base and the top, and compute `height = distance × (tan(angle_to_top) + tan(angle_to_base))`. Since LiDAR can't see the top of a mature tree, this is a separate tool, not a mesh fit:
- `CoreMotion` reads the device's tilt angle when the user sights the base and again when sighting the top.
- Horizontal distance comes from a user-entered value (matching current field practice — e.g. a measuring wheel) or an AR point-to-point tap when the base is within LiDAR range.
- A photo is taken from the sighting point as part of this step, tying the height reading to a visual record of where it was taken.
- The computed value pre-fills the height field; the surveyor can type over it directly if a manual reading differs — same as the existing gauge workflow.

**2. DBH / circumference — LiDAR circle-fit, with manual override.** Tap the trunk at breast height → raycast → slice nearby mesh vertices → Kasa circle fit → circumference in inches, matching the existing DBH field. The surveyor can overwrite the computed value by hand if the fit looks off or a tape measurement is preferred. This requires a device with a LiDAR-equipped camera — iPhone 12 Pro or later, or iPad Pro (2020 or later).

**3. Crown spread.** Two perpendicular AR point-to-point taps (canopy edge to canopy edge), reusing ARKit's standard ruler pattern — the same two-measurement convention used in the existing inventory, including its multi-branch rule (outermost branch tip to outermost branch tip, in two perpendicular directions).

**4. Species — manual entry.** A simple text field or searchable picker (common names), filled in once per tree as part of the same guided sequence, ideally with a photo already captured in step 1 to cross-check the ID later if needed. This is new functionality — no sensor involved, and no equivalent field exists in the prior dataset to override.

**Point cloud snapshot.** Store the filtered vertex slice already computed for the circle fit (not a full dense scan) as a small attachment on the tree record, so a reviewer can sanity-check or recompute the fit later without revisiting the tree.

### 5.2 Data model

A `Project` groups records and defines the site list surveyors pick from; every `TreeRecord` belongs to one project:

```
Project: id, name, siteCodes (list)

TreeRecord:
id, projectId, surveyorName, deviceId, timestamp, uploadedAt,
latitude, longitude, gpsAccuracy, utmNorthing, utmEasting, utmZone,
siteCode (selected from the project's siteCodes),
treeId,
dbhInches, heightFeet, spread1Feet, spread2Feet,
treeType (largeMatureTree | youngTree | newlyPlantedTree)  — derived from dbhInches ≥ 20 unless overridden,
isMultiBranch,
condition (good | fair | poor),
species,
notes,
photoURL, pointCloudSliceRef
```

Newark's pilot is one `Project` whose `siteCodes` are JH-W, JH-P, JH-A, JH-L, JH-K, JH-R, JH-UD, JH-E, PHIL, LEW, KELL; a new project just defines its own list. `dbhInches` and `heightFeet` are sensor-computed by default but directly editable (Section 5.1's manual override). `gpsAccuracy`, `utmNorthing`, `utmEasting`, and `utmZone` are computed automatically from `latitude`/`longitude` at capture time — `gpsAccuracy` from `CoreLocation`'s reported horizontal accuracy, the three UTM fields via a standard lat/long-to-UTM conversion — so a surveyor never enters them directly. `species` is a new, directly-entered field (Section 5.1, step 4).

**CSV column mapping** — export uses the raw export's exact headers (Section 1), not the Swift property names above:

| `TreeRecord` field | CSV header |
|---|---|
| `id` | `ec5_uuid` |
| `timestamp` | `created_at` |
| `uploadedAt` | `uploaded_at` |
| `id` (same value again) | `title` |
| `surveyorName` | `1_Surveyor_Name` |
| `latitude` | `lat_2_Location` |
| `longitude` | `long_2_Location` |
| `gpsAccuracy` | `accuracy_2_Location` |
| `utmNorthing` | `UTM_Northing_2_Location` |
| `utmEasting` | `UTM_Easting_2_Location` |
| `utmZone` | `UTM_Zone_2_Location` |
| `siteCode` | `3_Park_Location` |
| `treeId` | `4_Tree_ID` |
| `photoURL` | `5_Photo` |
| `dbhInches` | `6_Diameter_at_Breast` |
| `heightFeet` | `7_Tree_Height_ft` |
| `spread1Feet` | `8_Tree_Spread_tip_to` |
| `spread2Feet` | `9_Tree_Spread_tip_to` |
| `condition` | `10_Tree_Condition` |
| `notes` + `isMultiBranch` + `treeType` | `11_Notes_eg_mushroom` — `isMultiBranch` and `treeType` are both tracked as their own fields in-app but folded into the Notes text on export (e.g. "Large mature tree; multi-branch"), matching the sheet's existing style rather than adding new columns |
| `species` | `Species` |

`pointCloudSliceRef` is app-only — a debugging/QA artifact, not inventory data — so it's dropped from CSV export entirely. `photoURL` still matches the sheet's hosted-link pattern (a URL, not an embedded image) — photos taken in-app get uploaded somewhere reachable (e.g., Supabase storage) and the record just stores the resulting URL, the same way EpicCollect5 already works.

**Multi-surveyor visibility.** With surveyors assumed to have WiFi access in the field (Section 5.4), Supabase is the live source of truth: `treeId` is whatever the surveyor types in (matching site convention), and the Team screen shows what's already logged in near real time, so a second surveyor sees an existing Tree ID before creating a duplicate. This replaces the device-prefixed-ID or offline-reconciliation schemes that a no-connectivity assumption would otherwise require.

### 5.3 Screens (per Section 2's mapping)

1. **Projects (Home)** — list of projects, each showing its tree count and condition counters; tapping one opens its tree list. This is the app's landing screen.
2. **Project tree list** — counters by condition, then a tree card list (Tree ID + site, condition pill, DBH/height/last-measured/multi-branch, surveyor name), scoped to the selected project.
3. **Capture** — guided four-step sequence (Section 5.1): height sighting + photo, then DBH at the trunk, then crown spread at the canopy, then species entry, each step editable before moving on.
4. **Tree detail / edit** — header (Tree ID, site, condition), Measurements card, Inventory Information card (species included), "View Photos" / "Show on Map" actions.
5. **Map** — coordinate list now (interactive pins later), grouped by site code, sized by DBH, scoped to the selected project.
6. **Team** — which trees are already covered and by which surveyor, scoped to the selected project and synced from Supabase in near real time — the main tool for avoiding duplicate measurements across surveyors working the same project at once.
7. **Export & settings** — CSV export/share (Mail, AirDrop, Files), Supabase sync, surveyor name, project site-list management.

### 5.4 Persistence, export, and sync

- **Always-online assumption**: surveyors are assumed to have WiFi access in the field, so Supabase is treated as the live source of truth rather than a sync-when-convenient fallback — records sync in near real time, site lists come from project settings, and the Team screen reflects what's been measured as it happens.
- **Local storage**: records still persist on-device (SwiftData) as a resilience layer, so a momentary signal drop doesn't lose in-progress work — but sustained offline operation is no longer the primary case the design optimizes for.
- **CSV export**: serializes all records into a CSV using the sheet's exact column headers (Section 5.2's mapping table), then opens the iOS share sheet (Mail compose pre-filled with the CSV attached, AirDrop, Save to Files). Works independently of Supabase either way.
- **Supabase**: the primary backend, not an optional add-on — stand it up alongside the app design lock rather than later (see Next Steps):

```sql
create table projects (
  id uuid primary key default gen_random_uuid(),
  name text,
  site_codes text[],
  created_at timestamptz default now()
);

create table tree_records (
  id uuid primary key default gen_random_uuid(),
  project_id uuid references projects(id),
  surveyor_name text,
  collected_at timestamptz,
  uploaded_at timestamptz,
  latitude double precision,
  longitude double precision,
  gps_accuracy numeric,
  utm_northing numeric,
  utm_easting numeric,
  utm_zone text,
  site_code text,
  tree_id text,
  dbh_inches numeric,
  height_feet numeric,
  spread1_feet numeric,
  spread2_feet numeric,
  tree_type text,
  is_multi_branch boolean,
  condition text,
  species text,
  notes text,
  created_at timestamptz default now()
);
```

  A "Sync" action uploads local records and pulls others' records for the Team screen; CSV export works independently of Supabase either way.

### 5.5 Security & governance

This stays a local pilot: data lives on-device plus optional CSV export, with no PII beyond a surveyor name. Anything that becomes persistent infrastructure (hosting, backups, admin access) is out of scope for the app itself if and when this moves past pilot — that's an institutional decision to make separately, not something the app needs to manage.

## 6. Calibration Plan

Calibration doesn't depend on Newark's existing inventory or re-measuring any previously-logged tree — it's a fresh comparison on whatever trees are accessible to whoever's testing:

1. Pick ~10–15 trees spanning a range of trunk sizes (small to large mature), wherever the app is actually being tested.
2. Measure each one twice — once with traditional tools (diameter tape, tangent gauge, measuring wheel) and once with the app — and log the two sets of values side by side.
3. Define an acceptable error band (e.g., ±10% on DBH) before trusting the tool on new, unverified trees.

## 7. Build Architecture

**Where to build:** Claude Code, on a Mac with Xcode installed — not this planning session. This environment is sandboxed Linux with no Xcode, macOS, or iOS Simulator; it can produce source and schema designs but can't compile or run ARKit/RealityKit. The actual build needs a real compile/run feedback loop: `xcodebuild`, compiler errors, and a LiDAR-equipped test device (iPhone 12 Pro or later, or iPad Pro 2020+).

**Tooling split:** Xcode handles project scaffolding, building, and code review — it isn't writing the code. Claude Code, via the subagents below, authors all seven roles' code against the shared `TreeRecord` contract.

**Distribution for remote testing:** the mentor (Matt) is testing remotely, hours away, which rules out plugging his device into the build Mac directly. Rather than pay for an Apple Developer Program account ($99/yr) for TestFlight, the project shares the Xcode project via a GitHub repo; Matt clones it, opens it in his own Xcode, signs in with his own free Apple ID, and builds straight to his own device. This is free for both sides and avoids shipping devices back and forth, at the cost of Matt needing a Mac and a willingness to do a short one-time setup (clone, open, hit Run). The repo needs to stay buildable from a clean checkout (no machine-specific paths or untracked config) for this to work reliably.

**Repo:** [github.com/KieranButterfield/TreeInventory](https://github.com/KieranButterfield/TreeInventory) (private). Matt will need to be added as a collaborator, or the repo made public, before he can clone it.

**Subagent roles** (define as custom subagents in `.claude/agents/` in the project repo, dispatched by a lead session):

| Agent | Owns | Must respect |
|---|---|---|
| **Lead / Orchestrator** | Breaks the plan into tickets, locks the `TreeRecord` contract before others start, integrates branches | Section 5.2 data model |
| **AR-Capture Agent** | `ARViewContainer`, `Coordinator`, raycasting, Kasa circle fit, crown-spread tool | Emits a `TreeRecord`-shaped result |
| **Data & Persistence Agent** | SwiftData model, validation, tree-type derivation, multi-branch logic | Consumes/produces `TreeRecord` |
| **Export & Sync Agent** | CSV serialization, share-sheet/Mail integration, Supabase client + migrations, multi-surveyor sync (push/pull, Team-screen data) | Reads `TreeRecord` array, no UI dependencies |
| **UI Agent** | SwiftUI screens (list, capture overlay, detail, map, settings) | Calls Capture/Persistence only through their public interfaces |
| **QA/Calibration Agent** | Calibration harness, sample-tree comparison log, accuracy thresholds | Operates on real device builds, reports pass/fail |
| **Docs/Handoff Agent** | Protocol write-up, technical notes, user instructions | Documents the above, no code changes |

**Sequencing:** Phase 0 — Lead locks the `TreeRecord` contract and screen list. Phase 1 — AR-Capture, Data/Persistence, Export/Sync, and UI agents work in parallel against that contract. Phase 2 — Lead integrates and resolves interface drift. Phase 3 — QA/Calibration runs the field comparison (Section 6). Phase 4 — Docs/Handoff packages everything for handoff to the project team.
